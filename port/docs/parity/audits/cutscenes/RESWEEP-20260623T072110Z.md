## GAP 1 — case-insensitive alias matching missing → dropped dialogue line + dropped actor verb
- Behaviour: `objScript.interpretLineCommand` checks `lineWord1.char[last]=":"` FIRST → `#speakLine`
  regardless of case, and `translateScriptNameToObjCharacter` matches the alias case-INsensitively
  (Lingo string `=` is case-insensitive). The port matches aliases case-SENSITIVELY.
- Original: `casts/script_objects/objScript.txt:329-372` (`interpretLineCommand` / `translateScriptNameToObjCharacter`).
- Port: `port/src/data/cutscene.ts:119-125` — the say-regex `chars[say[1]!] !== undefined` and the
  actor-verb test `chars[toks[0]!] !== undefined` are exact-key lookups.
- REAL gap (named scene). In `casts/data/scr_stones1.txt` the line `M: Where might I find this spell?`
  parses as `{verb:"M:", arg:{symbol:"Where"}}` → hits `default: break` in the Thespian dispatch
  (thespian.ts:290) → **Merlin's line is silently dropped** (probe verb histogram for stones1 shows
  `"M:":1` as a command, and the SAY log skips that line entirely; 32 says vs the script's 33 dialogue
  lines). Also affects `casts/data/scr_demo_002.txt` line `T turnToFace m` → parses as global verb `T`
  → the TV never turns to face Merlin (the TV stays at its propAt facing). (demo_002 is not in the
  charter's named-scene list, but it is the same root cause and the same silent drop.)
- Fix sketch: fold alias keys to a case-insensitive lookup in the parser. e.g. build
  `const aliasOf = (w:string) => Object.keys(chars).find(k => k.toLowerCase() === w.toLowerCase().replace(/:$/,""))`
  and use it for both the dialogue test (`/^(\w+):/`) and the actor-verb test, mirroring
  `translateScriptNameToObjCharacter`'s case-insensitive `ListGetPosByProp` + trailing-`:` strip.

## GAP 2 — cutscene actor anim-char from `slice(0,3)` instead of the data `#name` → prestotolin invisible
- Behaviour: a cutscene actor's anim-sheet base is the actor's data `#name` (e.g. `presto`, `ber`, `uli`),
  exactly as `spriteCharOr` already resolves it for gameplay actors (anim.ts:38-44 uses
  `registry.resolveActor(name).name`). The Thespian uses a hardcoded `CUT_ANIM_CHAR` map and falls back
  to `name.slice(0,3)`.
- Original: actors render off their `#name` strip; `objScriptPerformer.createMissingPlayers`
  (`casts/script_objects/objScriptPerformer.txt:97-118`) spawns via `actorMaster.newActor` with the real
  actor type, so the sheet base is the resolved `#name`.
- Port: `port/src/scenes/thespian.ts:39-43` (`CUT_ANIM_CHAR` + `cutAnimChar`), used at thespian.ts:164.
- REAL gap (named scene). `prestotolin` has no `CUT_ANIM_CHAR` entry → `slice(0,3)="pre"`, but the bundled
  sheet is `presto_*` (there is no `pre_*`). Probe stones3 snap: `p` actor has `animChar:"pre"`,
  `hasSprite:false` — **Prestotolin is invisible for the whole stones3 chatter scene** (all his lines
  play with no on-screen sprite). `merlin`/`ulin`/`berlin`/`tv` happen to map correctly (mer/uli/ber/tv),
  so only prestotolin is broken among real scenes.
- Fix sketch: replace the map+slice heuristic with the same `#name` resolution `Anim.spriteCharOr` uses:
  `const dn = registry.resolveActor(name)?.name; if (dn && anims[dn+"_stand"]) return dn.replace(/^#/,"")` —
  then fall back to the existing heuristic. (Keeps mer/uli/ber/tv working and fixes presto.)

## GAP 3 — `setStage` leaves the background at the default tint instead of snapping to black
- Behaviour: `cutSceneMaster.setStage` (`casts/master_objects/cutSceneMaster.txt:283-288`) calls
  `backgroundColour(pSetSceneColour)` where `pSetSceneColour = rgb(0,0,0)` (init line 51) — an INSTANT
  black. Every full-stage script (intro/wasted/end/complete) opens with `setStage`.
- Port: `port/src/scenes/thespian.ts:411-416` (`setStage`) copies `this.bgTarget` into `this.bg`, and
  `bgTarget` defaults to `{10,16,24}` (thespian.ts:102) — a port invention.
- REAL gap. Probe (`setstage` mode): bg after `setStage` is `10,16,24`, not `0,0,0`. So the scene opens on
  a dark blue-grey, and the first `backgroundColourTo rgb(220,220,220)` tweens FROM 10,16,24 rather than
  from black — a wrong start colour + wrong tween baseline.
- Fix sketch: in `setStage`, set `this.bg = this.bgTarget = { r:0, g:0, b:0 }` (the `pSetSceneColour`
  black), matching `backgroundColour(pSetSceneColour)`. (Optionally also default `bg`/`bgTarget` to black.)

## GAP 4 — backgroundColourTo tween rate is per-RGB-step, not the fixed-duration percent tween
- Behaviour: `cutSceneMaster.backgroundColourTo` runs a colour transform at `speed=2`
  (`casts/master_objects/cutSceneMaster.txt:108-120`). The transform interpolates a percent 0→100 by
  `pSpeed` per frame (`objTransformer.txt:117-124` `VarToward(pCurr, pTarget, pSpeed)` with `pTarget=100`,
  `objTransColor.txt:22-43`), so it completes in `100/speed = 50 frames` REGARDLESS of colour distance.
- Port: `port/src/scenes/thespian.ts:490-495` (`tweenStage`) steps each channel by `±bgSpeed` RGB units
  per tick (`bgSpeed=2`), so duration scales with colour distance.
- REAL gap. Probe (`timing` mode): `backgroundColourTo rgb(220,220,220)` from the default bg reaches 220
  at tick 105, vs the original's fixed 50 frames. Long-distance colour changes (e.g. →220,220,220) are
  ~2× too slow; short ones are too fast.
- Fix sketch: model the tween as a percent 0→1 over `100/speed` frames (default 50), lerping
  `bgStart→bgTarget` per channel; the random-flash uses `speed=10` (10 frames) and the title uses
  `speed=4` (see GAP 5). Store `bgStart` at verb time and a frame counter instead of a per-channel clamp.

## GAP 5 — title fade duration 16 frames instead of 25 (colour-transform speed=4)
- Behaviour: `objCutSceneTitle.startColourChange` uses `speed=4` (`casts/script_objects/objCutSceneTitle.txt:
  startColourChange`), i.e. `100/4 = 25 frames` for each cross-fade leg (fade-down-to-bg, then reveal-to-
  gold; and the hide leg). Hold = `pDisplayTime = 150` frames (objCutSceneTitle.txt:init).
- Port: `port/src/scenes/thespian.ts:117-118` — `TITLE_FADE = 16`, `TITLE_HOLD = 150`.
- REAL gap (minor). `TITLE_HOLD=150` is correct; `TITLE_FADE` should be 25, not 16. Probe (`timing`):
  port title reaches full at t=16 and fades out over 15 frames; original would be ~25 each leg.
- Fix sketch: set `TITLE_FADE = 25`. (Note: the original cross-fades the title MEMBER COLOUR from the
  black background to gold, so on a non-black stage the text fades from black, not from transparent; the
  port's alpha cross-fade is an acceptable approximation on the black/dark open — leave as-is.)

## GAP 6 — lights (per-actor fade) duration 18 frames instead of 50; and no fade-in baseline
- Behaviour: `lightsUp/Down` → `objScriptPerformer.lightsChange` calls `startSlowFadeIn`/`startSlowFadeOut`
  on every player (`objScriptPerformer.txt:lightsChange`). `startSlowFade*` is a blend transform at
  `speed=2` over 0→100 (`casts/script_objects/modFader.txt:89-96,108-131`) → `100/2 = 50 frames`. The
  line completes when all faders finish (`pWaitingForPlayers→0`, `playerFaderFin`). `setStage`/
  `makePlayersInvisible` first sets every actor's blend to 0 (`modFader.invisible`, objScriptPerformer
  `makePlayersInvisible`), so `lightsUp` is a real 0→100 fade-IN.
- Port: `port/src/scenes/thespian.ts:34` `FADE_DURATION = 18` (used in `tweenStage`, thespian.ts:504),
  and `setStage` (thespian.ts:411-416) does NOT zero actor `alpha`, so actors are still at `alpha=1` when
  `lightsUp` runs → `lightsChange(1)` finds `fadeTarget===alpha` and the faders finish instantly (no
  fade-in). Probe wasted/end: alpha stays 1 throughout; `lightsUp` produces no visible fade.
- REAL gap. (a) Fade duration is 18 vs 50 frames; (b) `lightsUp` never fades in because `setStage` left
  actors lit. The completion-count gate itself is faithful (waitingForFaders mirrors pWaitingForPlayers).
- Fix sketch: set the per-actor fade step to `1/50` (or carry the `speed=2`→50-frame duration). In
  `setStage`, also set each actor `alpha=0` (mirroring `makePlayersInvisible`→`setBlend(0)`) so the
  following `lightsUp` fades them in from black. The `waitingForFaders` count gate stays as-is.

## teleport presentation (cosmetic — borderline gap, noting for completeness)
- `teleportInAt`/`teleportOut`: the original `modTeleport` (`casts/script_objects/modTeleport.txt:118-200`)
  plays a vertical-STRETCH + alpha-FADE (setSpriteHeight tall→natural + transBlend `#in` for in;
  natural→tall + transBlend `#out` for out) over the actor's CURRENT strip, then fires
  teleportIn/OutFinished. The port (thespian.ts:256-257) snaps the actor in/out instantly and sets
  `modeOverride="teleportIn"/"teleportOut"`, but there is no `*_teleportIn`/`*_teleportOut` anim sheet
  (probe: every teleport actor shows `action:"teleportIn"` resolving to `_stand`), and no stretch/fade is
  applied — so teleports are a plain pop-in/pop-out with no effect. Real in every chatter scene
  (stones1/3/4/5-10 all open with `teleportInAt`). Not counted in the headline GAPS list because it is a
  pure VFX flourish with no logic/placement effect, but a faithful pass would drive a stretch+fade here
  (the gameplay teleport already does this via the stretch death/teleport modules).
