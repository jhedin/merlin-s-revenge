## FINDING 1 (SS-1) — `small` bitmap font is rendered ONE CELL too far left (off-by-one) — HIGH IMPACT

- Original: `casts/script_objects/objFont.txt:57` `getLetterNum = StringGetPos(pLetterKey, theChar)`
  (1-based) → `objTileSet.txt:63 getTileNo` → `objDataMap.txt peekEntryNo` (tile N = sheet cols
  `(N-1)*cellW .. N*cellW`). Font key from `casts/data/fnt_small_properties.txt`.
- Port: `port/src/render/bitmapFont.ts:33` `index(ch) = key.indexOf(ch)` (0-based) and `:82`
  `drawImage(sheet, i*cellW, 0, ...)`.

### REAL gap (observed)
The committed `port/public/assets/fonts/05299_fnt_smallCC.png` (609×12, cellW=8) has a **leading BLANK
cell 0**: splitting into 8px cells, cell 0 is empty and the 60 glyphs ('A'..'?') occupy cells **1..60**.
So for `small` the glyph for key-index `k` lives at sheet-cell `k+1`, NOT `k`.

The other three faces start at cell 0 (verified): menu 'A'@cell0, numbers '1'@cell0, smallgrey 'A'@cell0.
Only `small` has the leading blank cell.

Browser probe output (per-glyph opaque-pixel count, port's `indexOf` addressing):
```
menu      "ABCabc" -> [41,52,32,37,42,28]   (all render)            OK
numbers   "12..90" -> [19,24,25,22,25,28,20,32,28,32] (all render)  OK
smallgrey "ABCabc" -> [20,23,15,17,16,12]   (all render)            OK
small     "ABCabc" -> [ 0,38,40,32,22,28]   <-- 'A' renders BLANK!  BUG
```
Because the port draws cell `indexOf(ch)`: `'A'`(0)→cell 0 = blank; `'B'`(1)→cell 1 = the 'A' glyph;
`'C'`→'B' glyph; … every letter is shifted to the PREVIOUS glyph and the final glyph ('?') never shows.

### Impact
`small` is the most-used HUD/UI face (`port/src/main.ts:456,457,488,496,507,516,523` — "Lv N",
"Kits N", "GMG", weapon symbol, control hints; `port/src/scenes/cutscenePlayer.ts:77,133,173,182,185,199`
— all cutscene captions; `port/src/scenes/menu.ts:59`; `port/src/scenes/weaponPalette.ts:86`). Every
one of these strings currently renders with the first character blank and each subsequent character
showing the wrong (previous) glyph — e.g. "GMG" → blank+"FL". Very visible.

### Fix sketch
Give `BitmapFont` a per-face cell offset and apply it in `index()`/`draw()`:
- `bitmapFont.ts`: add `cellOffset` (default 0); cell = `key.indexOf(ch) + cellOffset`; widen the
  fallback (return -1 only when `indexOf` is -1).
- `build_assets.ts` (h-block, ~line 366): detect the leading-blank cell per sheet, OR hard-set
  `cellOffset: 1` for `small`. Cleanest data-driven approach: in the build, compute the first
  non-blank 8px cell of each sheet (scan alpha after white/dark key-out) and emit `cellOffset` into
  `fonts[sym]`; `assets.ts:157` then passes it to the `BitmapFont` ctor.
- No new art to bundle — the sheet is already committed; only the addressing changes.
- (Root cause is almost certainly an extraction artifact: the original cast member's small sheet had
  no leading blank, so `StringGetPos`-1 lined up. The committed PNG gained a blank cell 0. Compensate
  with the offset rather than re-cropping, to avoid a Pillow dependency CI doesn't have.)

## FINDING 2 (SS-2) — rollover energy-bar fill uses the health gradient, should use the flat TEAM colour

- Original: `casts/script_objects/objMoveableEnergyBar.txt:81-82`
  `targetTeamColour = g.teamMaster.getTeamColour(pTarget.getTeam())` then
  `pEnergyBar.reset(energy, maxEnergy, targetTeamColour)` → `objEnergyBar.txt:106 setColour` paints the
  WHOLE fill that one flat team colour. Team colours: `casts/data/tem_*.txt #colour` e.g.
  goblins `rgb(0,255,0)`, blackSorcerer `rgb(100,100,100)`, ghosts `rgb(200,200,200)`,
  village/aldevar/monsterSummon `rgb(100,100,255)`, ninja `rgb(75,75,75)`, pitMonsters `rgb(115,99,67)`,
  ice `rgb(222,222,210)`, scarlet `rgb(100,100,100)`.
- Port: `port/src/render/rollover.ts:50` fills with `healthBarColour(frac)` — the red→yellow→green
  gradient from `port/src/render/healthBar.ts`.

### REAL gap
`healthBarColour` is the model for the PLAYER's own multicolour HUD bar (`objMulticolourEnergyBar`),
NOT the rollover bar. The rollover bar's colour encodes the unit's TEAM/allegiance and is constant
regardless of how hurt the unit is; only its WIDTH tracks energy (`objEnergyBar.updateEnergy`). The
port shows a unit at 50% HP as yellow and at full HP as green — wrong; the original would show, e.g.,
a goblin as solid green at any HP, a black-sorcerer unit as grey, etc.

### Other SS-2 sub-divergences observed (lower severity, same handler)
- Placement: original surround = `rect(left, bottom, right, bottom+4)` spanning the unit's WIDTH,
  sitting 4px tall just BELOW the unit (`objMoveableEnergyBar.txt:42-52 calcSurroundRect` +
  `modEnergy.txt:71 calcEnergyRectBottom`). Port draws a fixed 24px-wide box ABOVE the unit at
  `p.y-36` (`rollover.ts:42,49-50`). Width/position diverge.
- Inset/z-order are fine: original `barBorder=1` → fill inset 1px on a `dot` background, bar on top
  (`objEnergyBar.txt:124-131`); port draws a translucent black box then the fill on top — equivalent.

### Fix sketch
- `rollover.ts`: replace `healthBarColour(frac)` with the hovered unit's team colour. Bundle the
  team→colour map by reading `casts/data/tem_*.txt #colour` in `build_assets.ts`/`parse_data.ts`
  (emit `teamColours: Record<team, "r,g,b">` into `data.json`); resolve via the entity's team symbol.
  Pure data, no art.
- Optionally widen the bar to the unit's body width and move it below the unit to match placement.

### REAL gap (observed)

  original renders 8×8 per room (4×4 tile, sprite scaled ×2, contiguous).
- Faithful and verified OK: 5-state status set + `#cur` override + status priority
  (`getMiniMapData:169-171`, progression `[#clr,#inf,#fre,#spe]` in `structMaster.txt`); proximity
  blend `VarMapRange(minDist,[60,200],[10,90])` near→transparent (`minimap.ts:66` matches

### Fix sketch
- `minimap.ts`: anchor bottom-right (`oy = viewH - h*2 - …`), set `cell = tile.w * 2` (=8) and draw
  tiles `drawImage(img, px, py, img.w*2, img.h*2)` for the ×2 scale; apply `border` (1,1,2,2). No new
  art — tiles already bundled.
