## GAP 1 — Enemies / allies / CPU units gain NO max-energy on level-up (should grow 1%/level) — REAL, system-wide

**Mechanic:** per-level energy growth (`modEnergy.levelUpEnergy`). Every unit that levels (and every
unit DOES level — they kill each other) should raise its max energy by `energyIncPercentage`% of its
*base* energy each level.

**Original:**
- `casts/script_objects/modEnergy.txt:27` — `i[#energyIncPercentage] = 1` (the cast default).
- `casts/script_objects/modEnergy.txt:55` — `pEnergyIncAmount = params.energy * params.energyIncPercentage / 100`.
- `casts/script_objects/modEnergy.txt:169-181` — `levelUpEnergy`: `pMaxEnergy += pEnergyIncAmount; increaseEnergy(incAmount)`.
- Confirmed in disassembly: `extracted/engine/scripts/ParentScript 14 - modEnergy.ls:12` (`i[#energyIncPercentage] = 1`).
- Only **two** shipped actors override it: `act_dwelling` (-1) and `act_player` (2). Every other unit
  (swordOrc, archer, fireLizard, bat, golem, warrior/archer/monk allies, …) inherits the default **1**.

**Port:** `src/entities/archetypes.ts:420` —
`energyIncPercentage: num("energyIncPercentage", 0) || undefined`. The default is **0** (and `|| undefined`
even drops an explicit data `0`), so `Energy.init` (`src/components/combat.ts:25`) leaves `incPct = 0`,
and `Energy.levelUp` (`src/components/combat.ts:127-131`) computes `inc = 0` → max never grows.
Because `spawnAlly`/`spawnUnit` both delegate to `spawnEnemy` (`archetypes.ts:84-90, 97-105`), this hits
**every** non-player, non-dwelling unit. Only the player (`archetypes.ts:178`, default 2) and dwellings
(`archetypes.ts:144`, default -1) are correct.

**Observed (probe `_audit_census_levelling.ts`):**
```
swordOrc:  base=300  delta=0  CAST_expected(+1%)=3   *** GAP ***
archer:    base=200  delta=0  CAST_expected(+1%)=2   *** GAP ***
fireLizard:base=100  delta=0  CAST_expected(+1%)=1   *** GAP ***
bat:       base=50   delta=0  CAST_expected(+1%)=1   *** GAP ***
golem:     base=100  delta=0  CAST_expected(+1%)=1   *** GAP ***
player:    base=200  delta=4  CAST_expected(+2%)=4   OK
```

**Faithful behavior:** a unit with no `energyIncPercentage` in data should grow max energy by 1% of base
each level (heal by the same increment). A leveled-up swordOrc at L10 should have ~330 max HP, not 300.
This makes high-level enemies/allies meaningfully tankier — a core progression feel currently missing.

**Fix sketch:** change the enemy default to the cast value and stop discarding an explicit `0`:
`energyIncPercentage: typeof d["energyIncPercentage"] === "number" ? d["energyIncPercentage"] : 1`
(i.e. default 1, drop the `|| undefined`). Dwelling (-1) and player (2) already pass explicit values so
they are unaffected. No change needed in `Energy.levelUp`.

## GAP 2 — `levelUpEnergy` increment is rounded; cast is fractional (FLOAT) — REAL, minor

**Mechanic:** the per-level energy increment magnitude.

**Original:** `casts/script_objects/modEnergy.txt:55,173-178` — `pEnergyIncAmount = params.energy * pct/100`
with **no rounding** (Lingo float). A base-50 unit at pct 1 gains **+0.5** max per level; over 2 levels +1.0.

**Port:** `src/components/combat.ts:128` — `const inc = Math.round(this.baseEnergy * this.incPct / 100)`.
A base-50 unit (after GAP 1 is fixed) would jump **+1** per level (`Math.round(0.5)=1`) — double the cast
rate for small units, and it loses the smooth fractional accrual entirely.

**Faithful behavior:** keep the increment fractional (`this.baseEnergy * this.incPct / 100`, no round);
max/energy become non-integer, matching the cast. (Only surfaces once GAP 1 lets enemies grow at all; for
base ≥ 100 with pct 1 the value is already integral, so this is secondary.)

**Fix sketch:** drop the `Math.round` at `combat.ts:128`.

## GAP 3 — `getReward` floors `xp/2`; cast uses float — REAL, minor

**Mechanic:** kill XP reward.

**Original:** `casts/script_objects/modExperience.txt:112` —
`lastAttacker.gainExperience(pExperienceImWorth + (pExperienceGained / 2))` — float, **no floor**.

**Port:** `src/components/experience.ts:40` — `return this.imWorth + Math.floor(this.xp / 2)`. A victim
that had accumulated an odd amount of XP grants 0.5 less than the cast. Mostly hits when killing a unit
that had itself leveled (xp > 0); a fresh enemy (xp 0) is unaffected, so low impact.

**Fix sketch:** drop the `Math.floor` (`return this.imWorth + this.xp / 2`). NOTE: the killer's own
`gainXp` then accumulates a fractional `xp`, which is faithful — `pExperienceGained` is a float in Lingo.

## GAP 4 — Freeze duration capped at 1000 ticks; cast has NO upper bound on a single long freeze — REAL, low (edge-case)

**Mechanic:** freeze duration accumulation (`modFreeze.takeFreeze`).

**Original:** `casts/script_objects/modFreeze.txt:70-88` with the Counter helpers
(`casts/general_functions/{Counter,CounterSetCount}().txt`):
first hit sets `count = 999`, then `count -= freezeTime` where
`freezeTime = (|vx|+|vy|) * freezeMultiplier * 4`; each tick `count += 1`; defrost when `count >= 1000`.
So **duration = `freezeTime + 1`** ticks. `CounterSetCount` clamps only at the **ceiling** (1000) when
counting up — it does NOT clamp `count` from going far negative, so a single big hit (or stacked hits)
produces a freeze **longer than 1000 ticks** (e.g. two 800-magnitude hits → count −601 → ~1601 ticks).

**Port:** `src/components/freeze.ts:16,39` — `this.ticks = Math.min(FREEZE_MAX=1000, this.ticks + add)`.
The accumulator is clamped at 1000, so the longest possible freeze is 1000 ticks. The header comment
(`freeze.ts:3-6`) asserts the cast also caps at ~1000 "no matter how many hits land" — that is **incorrect**;
the cast caps the count *increase rate*, not the total thaw time.

**Observed (probe `_audit_census_freeze.ts`):**
```
freezeTime=999 : port-duration=999  | cast-duration=1000  OK
freezeTime=1500: port-duration=1000 | cast-duration=1501  *** GAP ***
freezeTime=4000: port-duration=1000 | cast-duration=4001  *** GAP ***
two 800-hits   : port ticks=1000    | cast duration ~= 1601  *** GAP ***
```

**Reachability / severity:** single shipped freeze weapons deliver modest magnitudes — resolved
freeze attacks (`_audit_census_freeze_real.ts`): `freezeBlast` (mult 3, powerScalar 0.25, charge 100),
`iceBoulder` (mult 3), `snowAura`/`quadAura`/`orcAura` (mult 0.5), `arcticBlast` (mult 1, powerScalar
0.75). A centre-hit `freezeBlast` is ~`(62−4)*0.25 * 3 * 4 ≈ 174` ticks — far below 1000. So the cap is
only reachable by **heavy stacking** (standing in overlapping aura mines / repeated bolts). Low severity,
but it is a genuine cast divergence and the explanatory comment is wrong.

**Fix sketch:** model the count exactly: track the cast `count` (`= 999 − Σ freezeTime`, no low clamp),
thaw when it reaches 1000. Equivalently raise/remove `FREEZE_MAX` and let `ticks` accumulate without the
1000 ceiling (duration = `freezeTime + 1`). Correct the `freeze.ts:3-6` comment either way.

## Verified FAITHFUL (no gap — checked and rejected)

- **0.5x freeze speed + thaw restore + teal gate** (`modFreeze.txt:32-42,70-81` vs `freeze.ts:30-63`):
  `freezeFactor()` is 0.5 while frozen, 1 after thaw; teal glow only arms when `attack.glowTeal` — all OK
  in the probe. (The port models speed as a multiplicative `freezeFactor` rather than mutating walkSpeed;
  numerically equivalent for the unleveled case the cast's `defrost` formula reduces to.)
- **Rising level threshold** `L³+L²+prev/(L+1)+5+init` (`modExperience.txt:87` vs `experience.ts:52,62`) —
  matches; default `init=0` means enemies level on their first earned XP, player init 10.
- **Strength + one-random-mana growth per level** (`modCharacterAttackProperties.txt:132-153`):
  strength always grows by `strengthIncLevel` (`control.ts:158,547`), one random mana stat grows
  (`mana.ts:33-41`) — present and correct.
- **Bullet-owner XP credit:** the cast skips reward when `lastAttacker.getType() = #objBullet`
  (`modExperience.txt:111`); the port already passes the bullet **owner**'s id as `attackerId`
  (`projectile.ts`/`splash.ts` → `combat.ts:44-46`), so the caster (not the bullet) is credited. Faithful.
