## 1. starMaster — CLEAN

### Cast behavior (`master_objects/starMaster.txt`)
Three spawn helpers, each spawning a different actor type:
- **`experienceStar(theObj)`** (`:11-24`) — spawns ONE `#experienceStar` actor at `theObj.getLoc()`, with
  `initVect point(0,-2)` (rises), placed one Z behind the unit (`setLocZ(locZ-1)`). Data
  `act_experienceStar.txt`: `#friction point(1,10)`, `#lifeCount 30`, `#inertia 50`, `#weight 1`. Triggered
  by `modStarReleaser.releaseStar` (`:39`) ← `modExperience.levelUp` (`:205-207`, `pReleaseStarOnLevel`
  default true) — i.e. **any unit that levels up** releases one rising star.
- **`markerStar(where)`** (`:27-31`) — spawns ONE `#star` actor, weight 0, `lifeCount 1` (one-frame marker).
- **`starBurstX(where, strength=3)`** (`:34-50`) — spawns FOUR `#star` actors at `where`, vectors
  `[(-1,-1),(-1,1),(1,-1),(1,1)] * strength` (a 4-way diagonal burst).

### Reachability of the two unaudited helpers — both DEAD in the shipped game
- **`markerStar`: zero callers** in the engine cast (`grep` over `casts/` finds the definition only; the only
  other copies are the editor build `extracted/map_editor/…starMaster.ls`). The `#star` actor type has **no
  `act_star` data file** in `casts/data/` — it is an editor-only actor. Confirmed faithfully-absent.
- **`starBurstX`: one caller — `objHair.txt:294`** (`setUpChain`, when a hair segment is grown). The hair
  subsystem is gated by `gPlayerHair`, and Merlin's `GameSpecific.ls:28` sets **`gPlayerHair = 0`**
  (the engine `main.ls` sets it to 1 only in the Rapunzel build). So `objHair.setUpChain` never runs in the
  shipped game → `starBurstX` is unreachable. Consistent with `_globals.md` (gPlayerHair=0 CLEAN) and the
  census's own hair-out-of-scope rows. The pre-existing note `code-vs-ts/_deathFlow.md:233` ("markerStar /
  starBurstX … out of scope") is **correct** (re-verified, not merely trusted — Charter §3).

### Port repro (`render/effects.ts`, `components/experience.ts:59-72`)
- The one reachable path is faithful: `experience.attemptLevelUp` sends `levelUp`, then
  `effects.spawnLevelUpStar(pos.x,pos.y)` — pushes a particle at the unit, `vy=-2` (initVect (0,-2)),
  `STAR_FRICTION_Y 0.9` per tick (vertical friction), `STAR_LIFE 30` (`#lifeCount 30`), drawn BEHIND the
  actor pass (setLocZ-1). Fires for **every** unit's level-up (not just the player), matching the cast.
- `forceLevelUp` (`experience.ts:50`) — the port's spawn-time level-grant (startLevel / armyMaster restore /
  dwelling) — deliberately does NOT spawn a star. This matches the cast: those paths run under
  `restoreFromArmyDetails`/`setupFromArmyDetails`, where `modExperience` sets `pReleaseStarOnLevel=false`
  (`:121-122`) during restore. Correct suppression, not a miss.

### Verdict: **CLEAN.** The only live spawn path (experienceStar) is reproduced with correct vector/friction/
lifetime/Z; `markerStar` (no caller, no `act_star`) and `starBurstX` (hair-only, gPlayerHair=0) are correctly
omitted as dead code. **No fix needed.** Suggested: leave the `_deathFlow.md:233` stamp as-is.

## 2. screenMaster — GAP (overlay-open does not pause the simulation; sub-screen input is swallowed)

### Cast behavior (`master_objects/screenMaster.txt`)
A screen FSM with a depth-1 back-stack: `pCurrentScreen` + `pBackToScreen` (each a `#screenCall`
struct = `{sym, caller}`). `goScreen` (`:53-59`) transitions and shifts current→back. `screenOn`
(`:159-168`) overlays a screen on top, again recording `pBackToScreen = pCurrentScreen`. `backAScreen`
(`:42-47`) does `screenOff(pCurrentScreen.sym)` then `screenOn(pBackToScreen.sym, caller)` — i.e. pops the
top overlay and re-shows the previous screen, returning control to its caller. The whole screen stack
(in-game menu + its sub-screens: showArmy / chooseKeys / instructions) is brought up over a game that
`gameMaster` has **paused** on escape; combat does not run while any of these screens is showing.

### Port repro (`scenes/sceneManager.ts`, `main.ts:266-329`)
Driven end-to-end via a throwaway FSM probe (title→intro→game→esc→open each overlay→back→resume; plus an
orphan sweep). **Back-stack is sound:**
```
esc1:                 overlay=ingameMenu  paused=true
openShowArmy:         overlay=showArmy    paused=FALSE   <-- BUG
backFromShowArmy:     overlay=ingameMenu  paused=true
... (keyConfig, instructions identical) ...
orphan-check showArmy/instructions/keyConfig: returns-to-menu = true (all)
```
No orphaned/unclosable screen; every overlay closes back to the in-game menu; `backAScreen()` clears the
overlay correctly. The depth-1 back-target is right (the cast's sub-screens all `backAScreen` to the menu,
which is exactly the port's `screenOn("ingameMenu")` close path).

### The GAP (taxonomy §1 cross-system seam + §3 TRUSTED divergence)
`SceneManager.isPaused()` (`sceneManager.ts:65`) is
`this.overlay === "ingameMenu" || this.inGameCut !== null`. It is **true only for `ingameMenu`**, NOT for the
K18 sub-screens `showArmy` / `keyConfig` / `instructions`. The main loop gates ALL simulation on
`if (scene.isPaused())` (`main.ts:319`). Consequence, confirmed by probe (`branch = LIVE-COMBAT`):

- With showArmy/keyConfig/instructions open, the loop **falls through `isPaused()` to the live-combat
  branch** (`main.ts:329+`): `rebuildCombatSubstrate()` + every entity `update()` runs. **Enemies move and
  the player can be killed while the reserve-army / choose-keys / instructions overlay is on screen.** The
  cast keeps the game frozen for the whole menu stack.
- The overlay's own input handler `screens.handleInput(ov,input)` (`main.ts:323-325`, inside the
  `isPaused()` block) is therefore **dead code** for these three overlays — so showArmy paging (←/→) and
  keyConfig OK/cancel/scheme-select (space/enter/↑/↓) **never reach the overlay**; those keys are consumed by
  the live game instead. Only `Escape` accidentally closes the overlay (it hits `main.ts:329`
  `escapePressed()`, which — overlay≠ingameMenu — re-opens `ingameMenu`, masking the bug as "esc works").
- Net: keyConfig **#ok (commit a scheme via space) does not fire from the overlay**, and showArmy is
  un-pageable from its own keys — both via this same root cause.

This is the §3 trap: the inline comment "the base stays paused; the overlay screen draws + handles input
until it closes" (`main.ts:267`) reads as acknowledged behavior but is **false** — `isPaused()` doesn't
cover these overlays, so neither the pause nor the input-routing it claims actually happens.

### Fix sketch (one line, `scenes/sceneManager.ts:65`)
```ts
isPaused(): boolean { return this.overlay !== null || this.inGameCut !== null; }
```
`overlay` is `"ingameMenu"|"showArmy"|"instructions"|"keyConfig"|null`, so `overlay !== null` makes every
open overlay pause the sim and route into the `main.ts:319` block (which already dispatches
`ingameMenu` vs the K18 sub-screens correctly at `:321/:323`). Both `isPaused()` callers (loop gate `:319`,
HUD "paused" label `:456`) want this. No other behavior depends on the current narrow semantics. A regression
guard belongs in a new `test/scene_manager.test.ts` (none exists): assert
`open(showArmy/keyConfig/instructions) ⇒ isPaused()===true`, plus the orphan/back-stack invariants this audit
drove by hand.

### Verdict: **GAP** — overlay-open does not pause; sub-screen input dead.

## 3. keyChooseMaster (+ objKeyBinding / objKeyDescriptions) — CLEAN (census premise was a cast misread)

### Cast behavior (`master_objects/keyChooseMaster.txt`)
The census framed this as a "key-rebind UI … rebind flow unverified." **The cast has NO per-key rebind.**
keyChooseMaster is a **keyset CHOOSER**: it loads the shipped key-sets (`retrieveKeyBindings` `:127-135`
pulls the `objKeyBinding` collection — the `bnd_*` data fields), shows the active set's control→key table
(`displayCurrentKeySet` `:83-100`), and lets the player pick a *different whole set*:
- `start` (`:142-146`): build the menu, `retrieveCurrentKeySet` (= `keyMaster.getCurrentKeySet`), display it.
- `menuOptionSelected` (`:106-121`): `#cancel` → `finish` + `screenMaster.backAScreen`; `#ok` →
  `keyMaster.setKeySet(pCurrentKeySet)` + `backAScreen`; any other → `changeSelection(set)` (preview only,
  no commit). There is no "press a key to bind a control" path anywhere in the object.
- `objKeyBinding` / `objKeyDescriptions` (`:1-11` each) are just `objTextData` subclasses — passive data
  holders for the `bnd_*` / description fields; no rebind logic.

Shipped key-sets (`casts/data/bnd_*.txt`): **arrow, mac, wasd, zqsd**. Decoding via `pKeysByNum`
(`keyChooseMaster.txt:35-61`): `bnd_wasd` = up13→`w`, down1→`s`, left256→`a`, right2→`d`; `bnd_zqsd` =
up6→`z`, left12→`q`, down1→`s`, right2→`d`.

### Port repro (`scenes/screens.ts:123-129,210-240`; `systems/input.ts`)
The keyConfig overlay is a **faithful keyset chooser, not a non-functional shell**:
- `SCHEMES = [both, arrows, wasd, zqsd]`; ↑/↓ cycle the highlighted scheme; the table previews
  `keyForControlInScheme(control, highlighted)` **without** mutating the live scheme (= `displayCurrentKeySet`
  preview). `space/enter` = `#ok` → `input.setScheme(scheme)` (= `keyMaster.setKeySet`) then close;
  `escape` = `#cancel` → close without committing. Exactly the cast's `menuOptionSelected` cases.
- **Commit takes effect in play AND persists:** `Input.setScheme` (`input.ts:60-63`) reassigns `this.keys`,
  which `moveVector()` (`:74-81`, read every gameplay frame) consumes — so a committed scheme changes
  movement immediately — and writes `localStorage["mr_scheme"]`, restored in the ctor (`:28`). Load-bearing,
  not cosmetic.
- wasd (w/s/a/d) and zqsd (z/q/s/d) **match the cast's decoded bindings exactly**. The port substitutes
  `both` (WASD+arrows convenience) and `arrows` for the cast's `arrow` + `mac`; `mac` is a Mac-keyboard
  variant — a minor content choice, not a flow gap.

### Caveat (shared with screenMaster GAP)
keyConfig's `#ok`/`#cancel`/scheme-select keys are currently swallowed by the live game because of the
screenMaster pause bug (§2) — but that is the screenMaster defect, not a keyChooseMaster logic defect. Once
`isPaused()` is fixed, `screens.handleInput("keyConfig", …)` runs and the (correct) commit fires.

### Verdict: **CLEAN.** The "rebind unverified / shell" census premise was a misread of the cast — there is no
per-key rebind to verify; the keyset chooser is fully functional, faithful to wasd/zqsd, commits, and
persists. No fix to keyChooseMaster itself. (It is gated behind the §2 fix to be reachable in play.)
