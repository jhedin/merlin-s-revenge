## VERDICT

- **Pathfinding routing/timing: CLEAN (faithful).** The beeline→scenic random-walk pather
  matches the cast, including the 5-frame stall constant and the (weak) wall-stick behavior —
  which is a faithful limitation of the *original* algorithm, not a port regression.
- **FSM transition TIMING: GAP.** The post-hit **reel/dazed window is a hardcoded flat 6
  frames** for every unit; the cast's reel ends dynamically when the knockback slide stalls
  (per-actor `frictionReel`-dependent). Driven: port daze 5–6f vs cast settle 18f (light) / 2f
  (heavy).

## CLEAN sections (driven, with numbers)

### C1. Open-field beeline — CLEAN
Enemy at (100,480), target at (700,480), no obstacles. Driven 200 ticks:
`d0=600 → dFinal=123`, `everScenic=false`, `maxYDrift=0`. The unit walks a straight
horizontal line and closes the distance — never needs scenic. Matches `updateBeeline`
(`modPathFinding.txt:97`) → `moveToLoc(pTargetLoc)`.

### C2. Stall constant = 5 frames — CLEAN
`pathFinding.ts:21 STALL_TIME=5` vs `modPathFinding.txt:36 pathFindingStallTime=5`.
Driven precisely (enemy walks into a full wall): per-frame delta trace shows the unit pins at
x=313 from frame 6; scenic engages on frame **11** — exactly 5 zero-movement frames later
(frames 6,7,8,9,10). Exact parity.

### C3. Stall-detector semantics — CLEAN (equivalent)
The static doc flagged a "subtle difference" (port reads position-delta; cast reads
`pMoveXY.getMoveVect()`). Verified equivalent: `objMoveXY.txt:201`
`pMoveVect = pSpr.loc - pOldSpriteLoc` is the **post-collision onscreen displacement** — i.e.
"did the unit actually move." The port's `|Δpos| > 0.01` (`pathFinding.ts:74-78`) measures the
same thing. **Ordering is also equivalent:** the cast runs the AI (`updateAI`) at the *start*
of `objMoveXY.update` (`objMoveXY.txt:162`), before integrating this frame, so it reads the
*previous* frame's move vector — and the port's CpuAI runs before Movement in the chain
(`archetypes.ts:79`), reading the previous frame's position delta. No off-by-one.

### C4. Waypoint generation `PointRoughly(loc, 100)` — CLEAN
`VarRoughly(var,100) = var − 100 + random(200)` (cast) ≡ `m.x + (rng.next()*2−1)*100`
(`pathFinding.ts:58`). Uniform ±100px per axis around the *current* loc. Exact.

### C5. seek-acquire → moveToAttack → attack-enter — CLEAN
Enemy at (200,480), target 60px away. Driven: first mode `moveToAttack` (acquired via
`teamMaster.findTarget` + commit), entered #attack at `attackDist=14` with `reach=16` — i.e.
it stopped and swung inside its melee reach (`targetInReachMelee`, `objAiCPU.txt:379`).

### C6. Cooldown gating between attacks — CLEAN
Point-blank, pinned target, 300 ticks: 34 attacks, **uniform 9-frame gaps**. The unit does not
attack every frame; the per-weapon cooldown counter gates re-entry (`control.ts:793`
`getCooldownFin`, reset on fire `control.ts:807`). Faithful to `objAiAttack.attack` (cooldown
gates entry, not the hit).

### C7. return-to-seek after losing target — CLEAN
Target killed mid-chase: within 60 ticks the AI returns to `findTarget`
(`updateMoveToAttack` drops the dead target → `goMode("findTarget")`, `control.ts:712`).
Driven `acquired=moveToAttack, targetDead=true, backToFind=true`. Matches
`objAiCPU.update #moveToAttack` `finState=#noTarget → goMode(#findTarget)` (`objAiCPU.txt:455`).

### C8. retarget throttle = 30 frames — CLEAN
`CpuAI.RETARGET=30` (`control.ts:480`) == `pRetargetCounter.tim[2]=30` (`objAiCPU.txt:24`).

### C9. death sequence — CLEAN (behaviorally)
A dead unit is governed by the dead-guard at the top of `update()` (`control.ts:660`):
`isDead → idle + return` *before* any mode logic, so a dead CPU is inert (no move/attack/seek)
regardless of its mode label. Driven: `isDead=true`, no real re-seek behavior. (The reported
mode value can read `moveToAttack` because the dead-guard short-circuits before the
`#die→dazed` mode flip runs — cosmetic only, the guard makes it inert either way.)

### C10. wall-stick is FAITHFUL, not a regression
Both the cast and the port use the same beeline→scenic *random-walk* pather (explicitly NOT
A*). Driven a wall between unit and target, a concave U-trap, and a full wall (real RNG, up to
5000 ticks): the unit pins on the wall face and never routes around (see "mechanism" below).
**The cast has the identical flaw** — `goScenic` picks a waypoint that is frequently *behind*
the wall (unreachable in x), the unit slides in y toward it, never re-stalls (it IS moving in
y), so it never re-rolls a fresh waypoint. The original's combat rooms are open arenas (per
census: a 2-tile solid border + per-tile exits, no interior obstacle mazes), so this never
bites in real play — the pather's only job is the border, handled by room transitions. Filed as
clean/faithful; see "Mechanism note" for the design caveat.

## GAP G1 — post-hit reel/dazed window is a flat 6 frames; cast reel ends on knockback-stall

**Mechanic:** when a CPU unit is hit it enters #dazed (frozen intent) and must hold that until
the reel/recoil completes, then return to seeking. The DURATION of that window is the audited
quantity.

**Cast (correct):** the top-down reel duration is **dynamic, physics-coupled, per-actor**:
- `modReel.txt:67` `goReelMode → goMode(#reel)`.
- `modReel.txt:78-80` on `#reel` entry: `frictionReel()` (per-actor knockback friction) +
  `resetStallCounter()`.
- `modReel.txt:161-167` `updateReel`: `fin = me.big.getStalled()` — **the reel ends when the
  knockback slide has stalled** (`objMoveXY.getStalled` = `pStallCount.fin`, i.e. the unit's
  onscreen movement decayed to ≤ `pStallSpeed` 0.2 for the counter window).
- `modReel.txt:99-104` `#reelFinished → goMode(#walk)`; the AI then leaves #dazed
  (`objAiCPU.characterModeChanged`, `objAiCPU.txt:116-135`).

So a unit that is shoved hard reels LONGER (it slides longer); a heavy unit with a high
`frictionReel` (boulderMonster 40, fourArmGolem 50 — vs the default 10) barely skids and reels
for only ~1–2 frames. The window is the knockback-settle time.

**Port (gap):** the dazed window is a **hardcoded flat 6 frames for every unit**, fully
decoupled from the knockback/`frictionReel`:
- `hurt.ts:50` `this.flashT = 6;` (set on every non-proof, non-lethal hit).
- `hurt.ts:28-32` `update()`: `flashT--`; at `flashT===0` it sends
  `characterModeChanged("#walk")`, which pops the AI out of #dazed (`control.ts:639`).
- `control.ts:634-640` `characterModeChanged`: `#reel → mode="dazed"`; recovery → `findTarget`.

There is no read of the knockback velocity or `frictionReel` anywhere in the reel-length path.

**REAL gap vs faithful — DRIVEN numbers** (`_audit_nav_fsm.test.ts`, real takeHit chain,
collision vector 40):
- A hard hit on goblinWarrior: **port stays dazed 5–6 frames; the knockback slide does not
  settle until frame 18.** The unit recovers and resumes moving/attacking while still visibly
  sliding ~3× longer than its daze. Cast reel here ≈ 18 frames.
- HEAVY vs LIGHT: with `frictionReel=50` the slide settles at **frame 2**; with `frictionReel=10`
  at **frame 18**. The **port gives BOTH a flat 6** — *too long* for the heavy (6 vs 2, a
  boulder that should barely flinch staggers 3× too long) and *far too short* for the light
  (6 vs 18). The entire per-actor `frictionReel` spread that defines reel length in the original
  is dropped.

This is failure-mode taxonomy **#3 (trusted documented divergence)** + **#8 (method-shape CLEAN
without a behavioral repro)**: the reel feedback was implemented as a fixed flash window and the
AI-side daze was tied to it, but the cast's reel length is the knockback-stall time, not a
constant.

**Fix sketch:** end the dazed window on the **knockback stall**, not a fixed counter — mirror
`updateReel = getStalled()`. In `Movement`, expose a "knockback settled" signal (e.g. the tick
`Math.hypot(kvx,kvy)` first drops below a stall threshold after a hit, matching
`objMoveXY.pStallSpeed`/`pStallCount` semantics). In `Hurt.update`, instead of decrementing a
flat `flashT=6`, hold the reel until that knockback-settle fires (with a small floor/ceiling for
units that take ~0 knockback or are reel-proof), THEN send `characterModeChanged("#walk")`. The
white-flash visual can keep its own short clock; only the **#dazed→recover** gate must move onto
the slide-settle so heavies snap back fast and hard-hit lights stay staggered the full slide.
(Keep the existing reel-proof short-circuit, `hurt.ts:49`.)

## Mechanism note (for C10 — faithful but worth a design ticket)

The scenic random-walk's wall-stick is faithful, but the port's documented "scenic exits on
arrival OR stall" optimization (`pathFinding.ts:46-47`, vs cast stall-only `modPathFinding.txt
:114`) does NOT help here: the waypoint chosen behind the wall is never *reached* (x blocked),
so the arrival exit never fires, and the unit never stalls (it slides in y), so neither exit
fires — it loiters on one waypoint indefinitely. Driven 1500 ticks at a full wall:
`distinctWaypoints=1, modeFlips=1` — the unit picks ONE waypoint and never re-rolls. Both trees
do this; it is only ever observable on interior walls the shipped maps don't contain. If a
future custom map adds interior obstacles, this is where units will get stuck — but as a faithful
reproduction it is out of scope for parity.
