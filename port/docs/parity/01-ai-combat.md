# Parity Audit 01 — AI Brains & Combat Engine

Domain owner: agent 1 (decision-making + combat resolution machinery). Read-only audit of the
Lingo source (`casts/script_objects/`, `casts/master_objects/`, `casts/general_functions/`,
`casts/data/`) against the port (`port/src/components/`, `port/src/entities/archetypes.ts`,
`port/src/engine/dispatch.ts`, `port/src/systems/`).

**Headline coverage for this domain: ~18%.**

The port is a hand-written *behavioral sketch* of one beeline melee enemy + a charged player blast,
not a port of the AI/combat machinery. The dispatch engine itself (`dispatch.ts`) faithfully
implements the four-primitive chain model from `PLAN_REVIEW.md`, but almost none of the actual
combat logic flows through it: the port reimplements combat as imperative `takeHit(dmg, attackerId)`
calls with **scalar damage**, abandoning the load-bearing `damage == knockback` semantics (`power`
is a `point`, knockback IS the damage). The 11 AI types collapse into one `EnemyAI` class with 4
hardcoded `kind` branches; the real hierarchical FSM (`findTarget → moveToAttack → attack`),
team-driven targeting (`teamMaster.findTarget`), relationship-based target tracking, weapon
manager, charge/cooldown model, and reel/recoil state machine are absent or faked.

## 2. Headline coverage estimate

**~18%** of the AI/combat domain by behavior.

- The **dispatch kernel** (`dispatch.ts`) is faithful to the four-primitive plan (chain walk / first-match / fold / full re-entry via `entity.send`) — but it's an empty stage: the combat that should run through it is reimplemented imperatively.
- **One** of 11 AI types (beeline melee CPU) is meaningfully present; 4 more are crude single-branch approximations; 6 are missing.
- The **combat resolution core** — `modAttack` knockback vectors, `teamMaster.impactMeleeAttack`, the `#attack` payload pipeline, `objMoveXY.vectAdd` knockback — is essentially **0%**. Damage is scalar; the defining `damage == knockback` semantic (`PORTING_PLAN §6`, `PLAN_REVIEW §1`) is not implemented anywhere.

## 3. Prioritized build targets (do in this order)

1. **`modAttack` attack object + `damage == knockback` resolution (`calcCollisionVect` → `modEnergy.takeHit`).**
   *Why first:* it's the foundation every AI and weapon sits on, and it's the plan's #1 fidelity risk. Until `takeHit` receives a **collision vector** (`point`) and `modEnergy` derives damage from `(|vx|+|vy|)·damageMultiplier` while `objGameObject.takeHit` applies that vector as knockback via `vectAdd`, the entire combat feel is wrong and no other combat module can be faithful. Concretely: change `takeHit(dmg, attackerId)` → `takeHit(collisionVect, attackingObj, owner)`; add `calcCollisionVect{Melee,Spell,Bullet,Splash}` (`modAttack`/`modSplashDamage`/`CollisionCalcVect`); apply knockback in Movement (`vectAdd` + `pInertia` damping from `objGameObject.takeHit`). Unlocks reel, freeze-time, recoil, splash — all of which read the vector.

2. **`teamMaster` + `findTarget` + relationship-based targeting (the `objAiCPU` FSM).**
   *Why second:* it unlocks essentially all enemy content and the multi-team army/dwelling combat. Build `pTeams` from `tem_*`, the unit-map broad-phase (`searchUnitMap` shell search), `findTarget(criteria/roles)`, `impactMeleeAttack` (the `calcAttackHit→calcCollisionVect→CallPayloadFunction` loop), and the `objAiCPU` mode FSM (`findTarget→moveToAttack→attack→attackFin`) with `#target` relationships + `#leaveGame` retarget. Replace the ad-hoc `nearestOfTypes`/`isFriendlyTeam`. This is the single biggest content unlock (every CPU enemy, allies, dwellings target through it). Depends on #1 (the resolution it calls).

3. **`modWeaponManager` + `objAiAttack` charge/cooldown from data (data-driven `#attack`).**
   *Why third:* it generalizes the player and gives enemies real, data-driven attacks (ranged/magic/melee with per-type cooldown = agility/dexterity/manaRegeneration, charge-max from mana capacity, attack-frame timing). Removes the hardcoded `SPELL`/`PUNCH` constants in `control.ts` and the `hasSword`/`hasSpell` booleans, replacing them with `pWeapons` + `setCurrentWeapon`/`selectSpell`. Lets weapon/spell **content** (agent 3's effect actors) plug into a real weapon slot. Depends on #1 (attack object) and benefits from #2 (target loc for ranged aim).

(After these three, the remaining AI variants — spellcaster kiting w/ bullet-dodge, ghost possession, bomber loop, chatter, builder/weaponseek — and `modReel`/`#recoil` physics, `objCharacter` mode FSM, splash, weapon technique fall out relatively cheaply because the substrate exists.)

## 4. Key faithfulness risks / subtle behaviors easy to get wrong

- **`damage == knockback` (the cardinal rule).** `modEnergy.takeHit`: `damage = (|cv[1]|+|cv[2]|) * damageMultiplier`; `objGameObject.takeHit` applies `cv` as a velocity add (`vectAdd`) damped by `(100-pInertia)%` *before* forwarding. So a heavy hit both hurts AND launches, and a glancing hit does neither. `act_*` `#power` is a **`point`** (e.g. `point(2,0)`) or a scalar for spells (`0.75`). The current scalar `takeHit(dmg)` throws this away — porting it as scalar damage permanently bakes in wrong combat feel and breaks reel/recoil/freeze (all read the vector magnitude/direction).
- **`takeHit` handler ordering is a contract.** `modExperience.takeHit` MUST run **before** `modEnergy.takeHit` (it sets `#lastAttacker` via `formRelationship` so XP is attributed before `loseEnergy → outOfEnergy → attributeExperience` fires). The port orders `Experience` before `Energy` in the archetype — good — but the mechanism is wrong: real XP attribution is event-driven (`#outOfEnergy → attributeExperience` reads the `#lastAttacker` relation, awards `imWorth + gained/2`, records the kill type), not the inline `gainXp` in `Energy.takeHit`. Bullets do NOT grant XP to themselves (`lastAttacker.getType() <> #objBullet`) — XP goes to the **owner**, not the projectile. The port passes `attackerId` (the bullet's owner id is passed as `ownerId` to `fireBullet`, so this is closeish) but skips the owner/`#objBullet` distinction.
- **`#recoil` blocks damage.** While `pMode = #recoil` (and `modEnergy.takeHit` early-returns on `getMode()=#recoil`), the unit takes **no** damage — a deliberate i-frame-by-state, distinct from `modInvince`. Enemies with `recoil:true` are much tankier. Port has neither recoil state nor the damage block.
- **Reel recovery is stall-based, not timed.** `modReel.updateReel` finishes when `me.big.getStalled()` (movement velocity below `stallSpeed` for N frames) — so a hard-hit unit reels *until it stops sliding*, then recovers. This couples reel duration to knockback magnitude and friction. The port's fixed 6-tick flash is unrelated.
- **`objMoveXY` stall detection is load-bearing well beyond movement.** It's the "done moving" signal for `modPathFinding` (beeline→scenic switch), bullet landing (`checkStalled`), `moveToTarget` arrival, and reel recovery. The port has no stall counter, so none of these state transitions can be faithful.
- **Targeting is committed + event-driven, not per-tick nearest.** `objAiCPU` forms an exclusive `#target` relationship and only re-evaluates on `pRetargetCounter` (30 frames) or `#leaveGame`/`#outOfEnergy` events. Re-scanning nearest every tick (current `EnemyAI`) makes enemies twitchy and ignores target priority/roles/team-override. `teamMaster.findTarget` also honors `targetCriteria` (`#lowestHealth` for healers!) and `targetRoles` priority — `healBlast` even rejects 100%-health targets (`objAiCPU.refreshTarget`).
- **Cooldown is mana-/stat-driven, not a constant.** `modWeaponManager.addCooldownCounter`: melee cooldown counter `inc = getAgility`, ranged `= getDexterity`, magic `= getManaRegeneration`. So higher stats = faster recovery. Player blast recast = `cooldown / manaRegeneration`. Charge max = `manaCapacity·chargeMaxModifier + chargeMaxBasic`, optionally scaled by `magicLimitMaster`. The port hardcodes these per-spell; they must come from `#attack` + character stats.
- **Spell is a live actor with a lifecycle, not an instant bullet.** `objAiAttack`: charging creates a spell actor (`ensureSpell`), grows it (`charge(amount, chargeLoc)`), and `releaseMagic` converts it to flying bullets (`release(targetLoc, spellSpeed)`). It follows the caster (`#updateReel` re-aligns it; `moveRoom` re-charges). `energyBlast` is `chargeMax:999`, `limitMagic:true`, `reach:9999` (so `moveToAttack` no-ops for casters). The port's instant `fireBullet` on release skips the charging-actor entirely (acceptable for the slice, but `chargePerUnit`/multistage summon spells can't be faithful without it).
- **`me.big.goMode` vs `ancestor.goMode` (full re-entry).** `modReel.goDamageMode → me.big.goMode(#reel)` restarts at the top of the chain so leaf `objCharacter.goMode` (anim/sound resets) AND `objCPUCharacter.goMode` (friction) AND `pAI.characterModeChanged` all run. The dispatch engine supports this (`entity.send`), but no combat code uses it yet; getting the reel/death/attack mode transitions to fire `characterModeChanged` on the AI is essential (it's how the AI reacts to being stunned: `objAiCPU.characterModeChanged` → `#dazed` AI mode).
- **`objAiFlyingBomber` does NOT suicide on contact.** The port's bomber self-destructs (`takeHit 999999`); the original runs a normal `moveToAttack→attack→moveToAttack` loop with a cooldowned attack. This is a behavioral invention to fix later.
- **`damageSpeed` threshold + death latch.** `modEnergy.takeDamage` ignores damage ≤ `damageSpeed` (armor-like). On death, `pEnergy = -100` latches so recovery can't revive. Missing in port (`Energy` just clamps at 0).
- **`getActorType = #characterEnergyRollOverMaster` special-cases targeting** in `teamMaster` (no team override for the mouse-over probe). A faithful `teamMaster` must replicate these probe carve-outs or the rollover and findTarget diverge.
